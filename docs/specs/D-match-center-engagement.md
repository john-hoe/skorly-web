# 方案 D — 比赛中心与互动增强（已与产品达成共识，待开工指令）

状态：**规划定稿（2026-06-12），未开工**。产品确认方向后执行。
背景：三期 S1 赛中阶段（见 `../phase-3-growth-revenue-plan.md`）。核心目标：**拉长赛中停留时长**（直接放大广告展示与留存），其次补齐赛后内容消费链路。
前置依赖：Spec B 实时链路已上线（KV 快照、每分钟直播 cron、`/api/live`、`fixture_events` 写入）；recap 自动化已上线（PR #50）；AI 竞猜手已上线（PR #51）。

## 已确认的产品决策

1. **文字直播：做**。模板化播报为主（零 LLM 成本、4 语言秒出），仅进球/红牌等大事件用轻量模型润色一句。
2. **动画直播：不做**。API-Football 无球权坐标数据，伪造动画损害信任；降级为**真实数据可视化**（控球/射门/势头条 + 进球动效）。
3. **视频集锦：只嵌入官方**。YouTube Data API 限定官方频道白名单自动匹配，绝不自托管（版权红线，沿用 news-pipeline 既定原则）。
4. **首页**：跟比赛节奏走（赛前推竞猜/赛中推直播间/赛后推 recap+集锦）；焦点卡可滑动；新增"今日看点"横滑条；**不做自动轮播**。
5. **测试环境**：轻量按需预览部署（独立 Worker + noindex），不建常驻 staging。

## 施工顺序（已确认）

```
D1 比赛中心（文字直播 + 数据可视化）   ← 最优先，吃赛中流量峰
D2 首页改版（可滑动焦点卡 + 今日看点条 + 动态区块顺序）
D3 官方集锦（接入比赛中心 + 赛果卡）
D4 「你 vs AI」对战榜
D5 预览部署机制
```

## D1 — 比赛中心：文字直播 + 数据可视化

### 架构原则（沿用 Spec B 不变量）
运行时 web Worker **不碰 DB**。所有直播数据由 jobs cron 写 KV，web 经 `/api/live/*` 只读 KV，客户端在直播窗口内轮询。

### D1.1 数据采集扩展（apps/jobs）
- `ingest-live.ts` 增加**实时技术统计**抓取：对每场 live 比赛每 ~3 分钟调一次 `/fixtures/statistics?fixture={id}`（射门/射正/控球率/角球/犯规等）。
- 配额数学：90 分钟 ÷ 3 = ~30 次/场 + 现有比分轮询与进球触发事件。按一天 4 场同时高峰估算 <400 次/日增量，叠加现有用量仍远低于 7500/日；沿用 `apiq:YYYYMMDD` 计数与三档降级（统计抓取在 80% 档位首先关停）。
- KV 快照 `live:fixture:{id}` 扩展字段：`stats`（最新统计）、`commentary`（最近 N 条播报）。

### D1.2 播报生成（模板为主 + 大事件 AI 润色）
- 新模块 `apps/jobs/src/live-commentary.ts`：
  - **触发源**：`fixture_events` 新事件（进球/红黄牌/换人/VAR）、比赛状态切换（开球/中场/终场）、统计里程碑（如控球率反超、射门数到 10）。
  - **模板播报**：每种事件 4 语言模板（参数：分钟/球员/球队/比分/统计值），无 LLM、毫秒级。
  - **AI 润色（仅大事件）**：进球与红牌额外调一次 qwen-flash 生成 1–2 句有情绪的解说（Worker 内直接 fetch LLM API；失败则只保留模板条目，soft-fail）。成本估算：~6 球/日 × 4 语言 ≈ 24 次轻量调用，可忽略。
  - **防编造**：润色 prompt 只允许引用传入的事件参数，不得新增事实（沿用 recap 的 facts-in-prompt 教训）。
- 存储：新表 `live_commentary`（`id, fixture_id, minute, sort_key, type, locale, text, created_at`，索引 `(fixture_id, locale, sort_key)`）。写 DB 的同时把最近 30 条放进 KV 快照。

### D1.3 比赛页改版（apps/web，保持 SSG）
- 直播窗口内，比赛页主体变为**比赛中心**：
  - 实时比分头部（已有 `live-match-header`）。
  - **文字直播流**：客户端轮询 `/api/live/[fixtureId]`，渲染 commentary 列表（新进条目带高亮动效；进球全屏闪光动效）。
  - **数据面板**：控球率/射门/射正/角球条形对比 + "势头计"（最近 10 分钟双方射门与进攻增速的简单加权，标注为"势头估算"）。
  - 完赛后该区域自动折叠为完整时间线，recap 与集锦（D3）上移。
- **SEO：`LiveBlogPosting` 结构化数据**。比赛日构建时（30 分钟一班）把已累积的播报作为 `liveBlogUpdate` 服务端渲染进页面 JSON-LD（`coverageStartTime/EndTime` 用开球/终场时间），抢"live"类搜索词。客户端实时性不受构建频率影响。

### D1.4 完成判据
- 直播中比赛页可见秒级更新的文字直播与统计面板，无需刷新页面。
- 进球后 ≤90 秒出现润色播报；模板播报 ≤15 秒。
- API 配额日用量增幅 <500；降级档位实测可关停统计抓取。
- Rich Results Test 验证 LiveBlogPosting 通过。
- 平均比赛页停留时长（PostHog）赛中时段 ≥3 分钟（基线对比）。

## D2 — 首页改版（跟比赛节奏走）

### 区块结构（移动端单列顺序为准）
1. **LIVE 顶条**（已有）。
2. **绿色横幅 + 焦点战卡**：
   - 多场同时直播时焦点卡**可左右滑动**（手动，圆点指示，不自动轮播）。
   - 卡片主按钮随状态切换：赛前="竞猜比赛"；直播中="进入直播间"（导去比赛中心）。
3. **"今日看点"横滑条（新增）**：6–8 张小卡，按时段自动选材——赛前：今日赛程/AI 预测；赛中：直播中场次；赛后：recap/集锦入口。数据全部来自构建时已有查询，无新后端。
4. **主栏动态顺序**：有直播 → 直播区块置顶（每场显示最近 2–3 条播报，点击进比赛中心）；无直播 → 即将开赛 → 最新赛果（卡片加"集锦 ▶"角标，D3 上线后生效）→ 最新文章。
5. **会员方案 teaser 下移**至文章区之后。
6. **侧栏**：排行榜区块追加 4 个 AI 竞猜手战绩行 + "你能赢过 AI 吗？"入口（D4 的首页钩子，D4 未上线前先链到排行榜页）。

### 完成判据
- 多场直播时焦点卡可滑动切换且每场数据实时；单场时与现状一致。
- 看点条在三个时段（赛前/中/后）选材正确；移动端横滑流畅无溢出。
- 首页 LCP 不回退（看点条图片懒加载）。

## D3 — 官方集锦（YouTube 白名单自动嵌入）

### 设计
- **白名单频道**（仅官方/持权方，初始集）：FIFA、各市场持权转播商（上线前人工核对各市场实际持权方后填入配置）。白名单存配置文件，增删走 PR。
- 新 job `apps/jobs/src/fetch-highlights.ts`（完赛后由 recap 同一触发链运行）：
  - YouTube Data API `search.list`（`channelId` 限定白名单 + 比赛双方队名 + 日期窗口）→ 命中视频 → 校验标题含双方队名 → 写入 `fixture_media` 新表（`fixture_id, kind='highlight', provider='youtube', video_id, title, published_at`）。
  - 配额：search 100 units/次，免费 10,000 units/日 → 每场 2–3 次检索（完赛后 1h/6h/24h 重试），远低于上限。需新增 env `YOUTUBE_API_KEY`（Google Cloud 免费创建，接入时引导）。
- 前端：比赛页（完赛态）与 recap 文章自动嵌入（复用 `social-embed.tsx` 的 YouTube 懒加载嵌入）；赛果卡显示"集锦 ▶"角标。
- 合规：只用 YouTube 官方 iframe 嵌入；视频被删时优雅降级隐藏；**绝不抓取/转存视频文件**。

### 完成判据
- 完赛后 24h 内 ≥80% 场次自动挂上官方集锦；嵌入源 100% 来自白名单频道。
- 失效视频自动隐藏不留空窗。

## D4 — 「你 vs AI」对战榜

- 排行榜页新增 tab："本周你 vs Skorly AI"：把 4 个 AI 竞猜手的周积分与用户对比，给战胜全部 AI 的用户发"AI 终结者"徽章（profiles 加 `badges` jsonb 或独立表，接入时定）。
- 个人中心显示"与 AI 对战记录"（同场预测对比：你 2-1 / AI·Elo 1-1 / 实际 2-1）。
- 分享图：复用现有 OG 图管线生成"我本周击败了 Skorly AI"卡片，社交渠道引流素材。
- 完成判据：榜单与个人对比可见、零额外采集成本（全部现有 predictions 数据）；分享卡可生成。

## D5 — 预览部署机制

- 新增 `pnpm cf:deploy:preview`：构建后部署到独立 Worker `skorly-web-preview`（workers.dev 域名），同一 DB（预览以只读浏览为主），并通过环境变量强制全站 `noindex` 响应头。
- 使用场景：高风险改版（如 D2 首页）先发预览 → 人工确认 → 再发生产。不建常驻 staging、不复制数据。
- 完成判据：一条命令出预览链接；预览站全路由带 `X-Robots-Tag: noindex`；生产部署流程不受影响。

## 风险与护栏汇总

| 风险 | 对策 |
|---|---|
| API-Football 配额（统计抓取增量） | 每 3 分钟/场 + 80% 档位首停统计；apiq 计数已有 |
| LLM 润色编造事实 | 只允许引用传入参数；失败回退模板条目（recap 教训已固化） |
| YouTube 配额/误匹配 | 白名单频道 + 队名校验 + 每场限 3 次检索 |
| 版权 | 视频仅官方嵌入；动画直播明确不做（无数据不造假） |
| 首页性能回退 | 看点条懒加载；改版走 D5 预览验证后再上生产 |
| 工期 | D1 赶小组赛中段上线价值最大；D2/D3 随后一周内；D4/D5 穿插 |
